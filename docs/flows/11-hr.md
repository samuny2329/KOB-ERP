## Data shape

```
hr.department         (tree)
  id, code (UNIQUE), name, parent_id, manager_id (FK to hr.employee, use_alter=true), active

hr.employee
  id, employee_code (UNIQUE), first_name, last_name, nick_name,
  email, phone, national_id,
  department_id, user_id (FK to core.user, optional),
  warehouse_id (where they work), hire_date, termination_date, active

hr.attendance     (clock-in / clock-out)
  id, employee_id, check_in, check_out
  worked_hours = (check_out − check_in) in hours (computed property)

hr.leave_type
  id, code (UNIQUE), name, days_per_year, requires_approval, active

hr.leave            (state machine)
  id, ref (UNIQUE), employee_id, leave_type_id, date_from, date_to,
  days, reason, state ∈ {draft, submitted, approved, rejected}, approved_by

hr.salary_structure
  id, code (UNIQUE), name, currency, active

hr.salary_rule        (line item that builds a payslip)
  id, structure_id, code, name, type ∈ {basic, allowance, deduction, tax},
  amount_fixed, amount_percent, sequence

hr.payslip           (one per employee per period)
  id, ref, employee_id, period_start, period_end,
  state ∈ {draft, computed, paid, cancelled},
  gross, net, total_deductions

hr.payslip_line     (resolved values of salary_rule for this slip)
  id, payslip_id, rule_id, name, amount
```

## State machine — Leave

```
draft → submitted → approved        (terminal)
            ↓     ↘ rejected        (terminal)
            └──── any non-terminal → cancelled (terminal)
```

## Happy-path flows

### A · Onboard

```
1. POST /hr/departments  { code: "WH", name: "Warehouse", parent_id?, manager_id? }
2. POST /hr/employees    { employee_code: "E-001", first_name, last_name,
                           department_id, user_id?, warehouse_id, hire_date }
```

### B · Clock-in / Clock-out

```
3. POST /hr/attendances  { employee_id, check_in: now }
4. PATCH /hr/attendances/{id}  { check_out: now }
   ── worked_hours computed at read time
```

### C · Leave request

```
5. POST /hr/leave-types  { code: "VAC", name: "Vacation", days_per_year: 10, requires_approval: true }
6. POST /hr/leaves
   { employee_id, leave_type_id, date_from: "2026-05-01", date_to: "2026-05-03",
     reason: "Family trip" }
   ──> Leave(state="draft", days=3)

7. POST /hr/leaves/{id}/transition?target=submitted
8. POST /hr/leaves/{id}/transition?target=approved
   ── approved_by = current user
   └── append_activity("hr.leave.approved")
```

### D · Payslip generation

```
9. POST /hr/salary-structures  { code: "STD", name: "Standard Monthly", currency: "THB" }
10. POST /hr/salary-rules
    { structure_id, code: "BASIC", type: "basic", amount_fixed: 25000, sequence: 10 }
    POST /hr/salary-rules
    { structure_id, code: "OT", type: "allowance", amount_percent: 10, sequence: 20 }
    POST /hr/salary-rules
    { structure_id, code: "SS", type: "deduction", amount_fixed: 750, sequence: 30 }

11. POST /hr/payslips
    { employee_id, period_start: "2026-04-01", period_end: "2026-04-30" }
    ── service:
       ├── for each rule in structure (ordered by sequence):
       │     resolved = rule.amount_fixed if rule.amount_fixed else
       │                rule.amount_percent / 100 * basic
       │     payslip_line(payslip_id, rule_id, name, amount=resolved)
       ├── gross = sum(basic + allowances)
       ├── total_deductions = sum(deductions + tax)
       ├── net = gross − total_deductions
       └── state = "computed"

12. POST /hr/payslips/{id}/transition?target=paid
    ── triggers a JE in the Bank journal (Phase 5 link).
```

## HRPage UI

- Department chips at top (filterable).
- Employee cards (name, code, department, hire date).
- Leave table with state badges + approve/reject actions.

## Hooks

- `Employee.user_id` is optional but, when set, links operational
  identity (warehouse worker = `core.user`) to HR identity.  Worker KPIs
  (`ops.worker_kpi`) are stamped with `user_id`; reports join through
  `Employee` to roll up by department.
- Approving a leave appends to `core.activity_log`.

## Differences vs Odoo

| | Odoo `hr.*` | KOB-ERP `hr.*` |
|-|------|---------|
| Employee → user link | `hr.employee.user_id` (optional) | same |
| Attendance | `hr.attendance` | same shape, simpler computed `worked_hours` |
| Leave types vs allocations | separate `hr.leave.allocation` model | not modelled — `days_per_year` lives on the type |
| Payroll | full localisation packs (TH/EU/US/...) per country | minimal Thai-style structure (basic + allowance + deduction + tax) |
| Salary structure | versioned w/ effective dates | not versioned — change requires creating a new structure |
